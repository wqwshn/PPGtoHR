### 问题一：突然跳变到160 BPM 并一直卡死

这是当前在线算法中最严重的问题，由两个致命 Bug 叠加导致：

**1. 谱峰追踪参数被错误赋值（参数类型 Bug）** 在离线算法中，寻找最近谱峰的函数 `Find_nearBiggest(Fre, prev_hr, rangeplus, rangeminus)` 期望接收的是一个**标量（Scalar）搜索半径**（例如 `0.3` Hz，表示允许寻找上下 18 BPM 内的峰）。 然而，在 Python 上位机 `matlab_worker.py` 中，初始化参数时错误地将频率边界当成了搜索半径赋值给了算法：

Python

```
# matlab_worker.py 错误代码：
"'HR_Range_Hz',[0.67,3.0]," 
"'HR_Range_Rest',[0.67,3.0]" 
```

这导致 MATLAB 内部执行 `Fre(i) - HR_prev < [0.67, 3.0]` 时，由于 MATLAB 的逻辑数组判定规则，它实际上**默认使用了数组的第一个元素 `0.67` Hz 作为追踪窗口**。`0.67` Hz 相当于 **40 BPM**！ **后果**：追踪窗口过于宽泛（±40 BPM），彻底摧毁了谱峰追踪（Peak Tracking）的意义。只要出现一个较高的运动噪声或倍频谐波（如 160 BPM），算法就会瞬间舍弃正确的基频并锁死在噪声峰上。

**2. 隔离状态下的历史轨迹“毒化”** 在线算法采用了软切换融合（`Fusion_HR = W * Freq_LMS + (1 - W) * Freq_FFT`）。 在静息状态下（W ≈ 0），系统输出 FFT 结果。但此时，**后台的 LMS 算法依然在强行运行并更新自己的历史记录 `Hist_LMS_HF`**。由于静息时没有有效的运动参考信号，LMS 实际上是在拟合噪声，其内部历史轨迹会迅速漂移并锁定到 160 BPM（结合 Bug 1）。 当人突然开始运动，W 瞬间变为 1，系统输出瞬间从 FFT 切换到 LMS，直接把已经“毒化”的 160 BPM 抛出来，**完美绕过了步长限制（Slew Rate Limit）**，导致心率出现断崖式跳变。

### 问题二：静息状态误差大且快速性较低 (Lag)

**1. 因果滤波引入的相位延迟（快速性低）** 离线算法在最后使用了 `smoothdata(..., 'movmedian')`，这是一个非因果操作（会利用未来的数据平滑当前点），所以没有延迟。而在线算法为了实时性，改用了一个容量为 5 的历史队列求中位数：

Matlab

```
% OnlineHeartRateSolver.m
obj.State.Smooth_Queue_HF = [obj.State.Smooth_Queue_HF, out_HF.Fusion_HR];
```

长度为 5 的中值滤波在实时流处理中会产生约 **2~3 秒的绝对延迟（Lag）**，导致您感觉心率变化“慢半拍”。

**2. 缺失了前置的离群点剔除（误差大）** 离线代码在重采样前有极关键的一步：`filloutliers(..., 'previous', 'mean')`。这能消灭微小抖动产生的毛刺。在线算法在重构时把这句话删除了，导致微小毛刺直接进入 FFT，破坏了静息段的频谱纯度。

------

### 💡 解决方案（代码修改报告）

为了修复上述问题，恢复算法原本的精度，您需要对现有的两个文件进行如下修改：

#### 1. 修改 Python 参数初始化 (修复跳变与卡死)

打开 `python/matlab_worker.py`，定位到 `_create_default_params` 函数（约 101 行）：

Python

```
# 【修改前】
"'HR_Range_Hz',[0.67,3.0],"
...
"'HR_Range_Rest',[0.67,3.0]"

# 【修改后】将其更正为标量搜索半径 (Hz)
"'HR_Range_Hz', 0.35,"     # 对应运动时允许 ±21 BPM 的追踪寻峰窗口
...
"'HR_Range_Rest', 0.15,"   # 对应静息时允许 ±9 BPM 的追踪寻峰窗口
```

*(注：如果想放大频谱惩罚的容忍度，可以将同位置的 `'Spec_Penalty_Width',0.1,` 修改为 `'Spec_Penalty_Width',0.2,`)*

#### 2. 修改 MATLAB 在线解算逻辑 (修复历史毒化与延迟)

打开 `matlab/OnlineHeartRateSolver.m`：

**A. 恢复离群点剔除 (约 105 行 `compute_single_path` 内)：**

Matlab

```
% 【修改前】
ppg_col   = double(raw_win(:, 1));

% 【修改后】
ppg_col   = filloutliers(double(raw_win(:, 1)), 'previous', 'mean');
```

**B. 增加历史交叉对齐逻辑，防止静息转运动瞬间的突变 (约 185 行，`Fusion_HR` 计算之后)：**

Matlab

```
Fusion_HR = W * Freq_LMS + (1 - W) * Freq_FFT;
            
% 【新增代码】交叉状态对齐，防止隐藏路径的轨迹漂移毒化
if W > 0.8
    % 强运动态：强迫 FFT 历史跟随 LMS，防止切回静息时突变
    obj.State.(['Hist_FFT_' path_type]) = Freq_LMS;
elseif W < 0.2
    % 强静息态：强迫 LMS 历史跟随 FFT，防止切入运动时突变到 160
    obj.State.(['Hist_LMS_' path_type]) = Freq_FFT;
end

out.Fusion_HR = Fusion_HR;
```

**C. 缩短平滑队列以提高快速性 (约 67 行 `process_step` 内)：** 将原有的 `Smooth_Win_Len` 判断上限强行调小，或修改 Python 传入的参数。在 `OnlineHeartRateSolver.m` 中：

Matlab

```
% 【建议修改】将平滑队列的维护长度从默认的 5 降低到 3
if length(obj.State.Smooth_Queue_HF) > 3 % 原为 obj.Para_HF.Smooth_Win_Len
    obj.State.Smooth_Queue_HF(1) = [];
end
if length(obj.State.Smooth_Queue_ACC) > 3
    obj.State.Smooth_Queue_ACC(1) = [];
end
```

**总结：** 执行上述修改后，参数寻峰逻辑将恢复正常运转，不仅能彻底解决从 67 徒增到 160 BPM 并卡死的问题，还能将静息状态下的响应速度（快速性）提升约 2 秒，达到与离线算法几乎一致的表现。
